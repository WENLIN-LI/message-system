RoomTalk Interview Prep

A junior-engineer-friendly explanation of the project, its architecture, and the interview talking points.

1. Simple Pitch

30-second version:

RoomTalk is a realtime AI collaboration app. People can join shared rooms, chat with each other, ask AI questions, upload private media, and use AI coding agents inside safe sandboxed workspaces.

The Node.js server controls users, permissions, messages, AI turns, cost tracking, storage, and sandbox lifecycle. The actual code execution happens in E2B sandboxes, so untrusted commands do not run on the production server.

One-sentence version

RoomTalk is a shared chat and coding workspace where humans and AI agents collaborate in realtime, with code execution isolated inside per-room E2B sandboxes.

What makes it interesting

  • It is not just a chatbot. It is a shared room where multiple users can see the same conversation and AI work.
  • It is not just a code editor. It can run an AI coding agent inside a sandboxed workspace.
  • It separates control from execution: Node.js coordinates the system, while E2B runs untrusted code.

3. Feature Breakdown

Realtime Chat

Users join rooms and exchange messages live. When a message is sent, the server saves it and broadcasts it to everyone in the room.

Junior explanation: Socket.IO lets the server push updates to browsers immediately, instead of waiting for the browser to keep asking for new data.

User sends message
  -> browser emits socket event
  -> server validates access
  -> server saves message
  -> server broadcasts message to room

Multi-Provider AI Streaming

RoomTalk can call different AI providers and stream the answer back as chunks. The user sees the answer appear while it is still being generated.

  • ask_ai starts an AI request.
  • ai_chunk sends partial text to the browser.
  • ai_stream_end marks the final saved response.

Junior explanation: The app does not wait for the full AI answer. It shows pieces as they arrive, then saves the final result.

Code-Agent Rooms

A code-agent room is a chat room connected to a sandboxed coding workspace. Users can ask Coco or Codex to inspect files, edit code, run tests, and start applications.

Normal RoomCode-Agent Room
Chat with people and AIChat plus a sandboxed workspace
AI answers questionsAI can edit files and run commands
Messages are the main outputMessages, tool events, diffs, terminal output, and previews are all part of the workflow

Browser IDE

The browser workspace gives users a lightweight coding environment without installing anything locally.

File Browser and Editor

Users can browse workspace files and edit source code from the web app.

Git Diff Viewer

Users can inspect what changed, similar to reviewing a pull request.

Line-Level Review Comments

Users can comment on specific lines in a file or diff and send that feedback to the agent.

Artifacts View

Users can see and manage published static outputs.

PTY Terminal

A PTY terminal is an interactive terminal displayed in the browser but connected to a real shell running inside the sandbox.

Users can run commands such as:

npm test
npm run dev
git status
Ctrl+C

Junior explanation: It feels like a normal terminal, but the commands run in the E2B sandbox, not on your laptop and not on the production server.

Dev-Server Previews and Static Artifacts

If the sandbox starts a dev server, RoomTalk can show the running app inside the browser. If the user publishes static output, RoomTalk stores it separately so it keeps working even after the sandbox expires. 

npm run dev
  -> app starts in sandbox
  -> RoomTalk opens a browser preview

npm run build
  -> static files are published
  -> stable /p/... URL keeps working later

4. Architecture

Control Plane vs Execution Plane

LayerMeaningExamples
Control Plane The part that makes decisions and enforces rules. Node.js server, permissions, room state, tokens, message persistence.
Execution Plane The part that runs code and commands. E2B sandbox, shell commands, tests, dev servers, agent tools.

This separation is important because production servers should not directly run untrusted user or AI-generated code.

Redis/PostgreSQL Composite Storage

RoomTalk uses PostgreSQL and Redis for different jobs.

StoreUsed ForSimple Analogy
PostgreSQL Rooms, messages, members, media records, auth, durable agent turns. Long-term memory.
Redis Online presence, socket sessions, member counts, cache. Fast temporary memory.
CompositeRoomStore One interface that delegates to the right store. A receptionist that sends each request to the right department.

Private Media Pipeline

Media files are stored in object storage such as Tigris/S3-compatible storage. The server gives the browser signed URLs for upload and download. 

Browser asks for upload URL
  -> server checks room access
  -> server returns signed upload URL
  -> browser uploads file to object storage
  -> browser confirms upload
  -> server saves MediaAsset record
  -> future downloads use signed read URLs

Why this design? Large files do not need to pass through chat messages, and private files are not public by default.

Sandbox Lifecycle

Each code-agent room can connect to one E2B sandbox. The lifecycle service creates, reconnects, pauses, resumes, expires, or migrates that sandbox.

  • Create: make a new sandbox when a room needs one.
  • Reconnect: reuse an existing sandbox if it is still alive.
  • Active TTL: keep the sandbox alive longer while an agent turn is running.
  • Idle TTL: shorten the lifetime after work is done.
  • Migration: move workspace files when the pinned sandbox artifact changes.

Reusable JSONL Daemon

A daemon is a long-running process. JSONL means one JSON object per line. RoomTalk uses a reusable daemon inside the sandbox to avoid starting a new runner from scratch for every agent turn. 

Start daemon once in sandbox
  -> send JSONL request for turn 1
  -> receive JSONL events
  -> send JSONL request for turn 2
  -> receive JSONL events

Example request:

{"schemaVersion":1,"type":"run","roomId":"room-1","turnId":"turn-1","prompt":"Fix the login bug"}

Example events:

{"schemaVersion":1,"type":"status","status":"running","turnId":"turn-1"}
{"schemaVersion":1,"type":"text_delta","messageId":"msg-1","delta":"I'll inspect the code."}
{"schemaVersion":1,"type":"tool_call","turnId":"turn-1","name":"read_file"}
{"schemaVersion":1,"type":"tool_result","turnId":"turn-1","content":"Found the auth handler."}
{"schemaVersion":1,"type":"final","turnId":"turn-1","answer":"Fixed the login bug."}
Important nuance:

Reusable does not mean multiple agent turns edit the same workspace at the same time. The server and daemon both enforce one active turn per room/workspace.

5. Technical Challenges and Solutions

This section restores all 18 challenges from the original guide. For an interview, explain each one in three steps: what went wrong, what you changed, and why the change is useful.

Code-Agent Challenges

Challenge 1: Turning a Chat Room into a Shared Execution Environment

Problem: A coding agent must edit files, install packages, run tests, and start servers. Running those commands inside the production Node process would be dangerous.

Solution: RoomTalk separates the control plane from the execution plane. Node checks permissions and records the turn; one E2B sandbox per room runs the code.

Interview takeaway: The key decision was ownership. Node decides who may do something, while E2B performs the untrusted work.

Challenge 2: Keeping AI Text and Tool Events in the Correct Order

Problem: An early runner streamed AI text immediately but saved tool calls later. After refresh, the transcript could show a conclusion before the command that produced it.

Solution: The engine emits text, tool-call, and tool-result events in their real execution order. Node assigns a server-side position and starts a new AI text segment after each tool boundary.

Why the frontend cannot fix it: If two text segments were already merged, the browser no longer knows where the tool belongs. Ordering must be captured by the layer that first sees the real sequence.

Challenge 3: Sharing Room Context with Least Privilege

Problem: The agent needs earlier room messages, but giving it the full database or a long-lived token would expose too much data.

Solution: Each turn receives a short-lived scoped token. The message-system CLI requests only bounded history, search results, or individual messages through a local broker. The server checks room, client, turn, and mode again before returning safe fields.

Interview takeaway: Model access, room context, and publishing use separate capabilities, so one leaked token cannot unlock everything.

Challenge 4: Making the Workspace Reviewable Without Bypassing Security

Problem: Users need to inspect files, diffs, terminals, and previews, but a direct browser-to-E2B connection would bypass Node identity and permission checks.

Solution: Authenticated Socket.IO handlers broker every workspace operation. Files and diffs use bounded snapshots; PTY and preview features use long-lived sessions and delta events; large media uses controlled asset routes.

Interview takeaway: Git refs make diffs repeatable, and line-level review comments become structured context for the next agent turn.

Challenge 5: Long Tasks, Reusable Daemons, and Sandbox Lifecycle

Problem: Starting a new runner for every turn is slow. Long tasks, terminals, and dev servers must survive between tool calls, while stale processes must not leave a room stuck as running.

Solution: One reusable daemon processes turns serially in each sandbox. Active work extends the sandbox TTL; idle work returns to a shorter pause timer. Startup cleanup, bounded waits, shutdown hooks, and health checks recover stale state.

Upgrade path: When the pinned sandbox artifact changes, RoomTalk exports the workspace, creates the new sandbox, imports the archive, initializes Git, and swaps the sandbox ID.

Frontend Challenges

Challenge 6: Media Viewer Gesture Engine

Problem: Tap, double-tap, pinch zoom, pan, horizontal paging, and swipe-down dismissal all begin with similar touch events.

Solution: A gesture state machine starts in tap, then locks to one mode after movement passes a threshold. Direction ratios prevent a diagonal swipe from changing modes halfway through.

Important details: Horizontal and vertical intent use slightly different direction thresholds. Double-tap requires two taps within 220 ms and 34 px. Dismissal accepts either enough distance or enough velocity, while pinch zoom keeps the midpoint between the fingers visually fixed.

Performance: Three pending channels hold paging, dismissal, and image transforms. One requestAnimationFrame callback flushes them together instead of writing to the DOM on every touch event. The original implementation also coordinated three visual layers: controls fade first, the backdrop becomes transparent, and the image shrinks.

Challenge 7: Three-Tier Media Cache

Problem: Presigned URLs change over time, so normal browser URL caching may download the same private image again.

Solution: Cache by stable assetId: check an in-memory Object URL first, then the browser Cache API, then the network. Concurrent requests share one Promise, preventing several components from downloading the same file at once.

Eviction: An LRU-style policy removes the oldest entries when the cache passes its 300 MB limit and revokes related Object URLs.

Extra detail: Video poster frames are generated with canvas and cached too.

Challenge 8: Mobile Viewport and Text Input

Problem: iOS and Android resize pages differently when the keyboard opens. A fixed 100vh can hide the input, and pressing Enter during Chinese or Japanese IME composition can send an unfinished message.

Solution: Use the Visual Viewport API to track the usable height and update CSS variables. A ratchet-style baseline changes only when the keyboard is closed or the viewport grows, which avoids treating keyboard shrinkage as a new normal height.

IME guard: Check composition event state, the native event flag, the key code, and a short 80 ms grace period before treating Enter as send.

Challenge 9: Sticker System

Problem: The collection contains 2,275 stickers. Loading and rendering every image at once would create too many DOM nodes and network requests.

Solution: Render only the current page and its immediate neighbors, then preload adjacent pages so swiping feels instant. A 350 ms long-press timer opens a preview and suppresses the following click so previewing does not send the sticker.

Interview takeaway: The design balances fast first paint, smooth browsing, and low memory use.

Challenge 10: Message List UX

Problem: Waiting for the server before showing a sent message feels slow. Loading older history can also jump the user's scroll position.

Solution: Use a three-phase optimistic state: local pending message, confirmed server message, or failed message that can be retried. When prepending older messages, compare scroll height before and after and add the difference to scrollTop.

Auto-scroll: A ResizeObserver keeps the view attached to the bottom only when the user was already near the bottom, so reading older messages is not interrupted.

Interview takeaway: Realtime correctness and perceived speed are separate problems; the UI must handle both.

Backend Challenges

Challenge 11: Mobile WebSocket Recovery

Problem: Mobile browsers suspend background tabs, drop sockets, and sometimes keep an old connection alive briefly. A simple reconnect can miss the room or double-count the same user.

Solution: Persist the intended active room in the browser, re-register identity after reconnect, and rejoin idempotently on reconnect, visibility, online, and page-restore events. Recovery is layered because no single browser event fires reliably on every phone.

Duplicate protection: Presence counts unique clientId values instead of raw socket IDs, so a briefly overlapping old and new socket still represent one member.

Challenge 12: Redis to PostgreSQL Migration

Problem: Redis was useful for the first version, but long-term messages, membership, and auth need relational durability and query support.

Solution: Define one durable store contract, implement it in both Redis and PostgreSQL, choose the implementation through PERSISTENCE_STORE, and migrate incrementally with rollback available.

Interview takeaway: The migration changed the storage implementation without forcing socket handlers to know which database was active.

Challenge 13: Room State Synchronization and Consistency

Problem: Multiple clients can rename a room, change its password, or update speaking hours while another client still holds an older room object.

Solution: The server returns a complete room object instead of a partial merge. A monotonic room_version tells the client which state is newer, and the write acknowledgement immediately returns the latest object for read-your-write behavior.

Why not CRDTs: RoomTalk has a central online server and mostly server-owned operations. CRDT complexity is not justified unless offline peer editing becomes a real requirement.

Challenge 14: Multi-Provider AI Abstraction

Problem: OpenAI, Anthropic, DeepSeek, and OpenRouter use different request formats, stream event shapes, model names, and usage fields.

Solution: Normalize models and provider clients behind one server interface. Convert provider-specific chunks into RoomTalk events, apply provider-specific rate handling in the adapter, and record usage and pricing in one internal format.

Interview takeaway: Provider-neutral persistence allows an interrupted stream to recover consistently even if the selected provider changes.

Challenge 15: Redis Lua for Atomic Operations

Problem: A read-modify-write sequence can interleave with another request, causing lost updates, duplicate membership, or incorrect counters.

Solution: Move related Redis checks and writes into Lua scripts. Redis runs each script atomically, so another command cannot interrupt halfway through. Scripts cover operations such as room-version increments, member counts that deduplicate sockets by client ID, and deleting a message while recomputing room activity.

Reference counting: A member remains online until all sockets belonging to that client have left, which handles multiple tabs and reconnect overlap.

Junior analogy: Instead of making several trips to the bank counter, submit one transaction that either completes as a unit or does not happen.

Challenge 16: Store Contract Tests for Two Implementations

Problem: RedisStore and PostgresStore can compile against the same interface but still behave differently for edge cases.

Solution: Run the same behavior tests against both stores. The contract checks results, errors, pagination, ordering, and idempotency instead of testing only database-specific code. In-memory Redis and PostgreSQL test doubles simulate Lua and SQL behavior so the same scenarios can run quickly.

Challenge 17: Message Cache Version Consistency

Problem: A Redis cache in front of PostgreSQL can return old message pages after an edit, delete, reaction, or new message.

Solution: Use a version-aware read-through cache. CompositeRoomStore invalidates the room cache only after successful writes that can change message results, including edits, deletes, reactions, and streamed AI appends. PostgreSQL remains the source of truth; a cache miss affects speed, not correctness.

Challenge 18: CJK-Aware AI Context Budgets

Problem: AI prompts must fit a model's context window. English words and Chinese, Japanese, or Korean characters do not map to tokens in the same way.

Solution: Use a coarse-to-fine selection process: reserve space for the system prompt and answer, estimate each message with language-aware rules, then keep the newest useful history that fits. Mixed CJK and Latin text is estimated separately instead of using one characters-per-token ratio.

Interview takeaway: Estimation protects reliability; usage accounting provides the final cost truth.

Deeper Implementation Notes

The cards above are the short answers. Use the following details when an interviewer asks how the design actually works.

Code-Agent Turn Flow

Browser sends authenticated prompt
  -> Node checks global rollout and room-level access
  -> CodeAgentSessionService creates a durable turn
  -> sandbox lifecycle creates, reconnects, or resumes E2B
  -> Node issues separate context/model/publish capabilities
  -> reusable daemon routes the request to Coco or Codex app-server
  -> ordered JSONL events return to Node
  -> Node persists each event position and broadcasts it to the room
  -> final usage, cost, status, and backend session ID are saved

Why Ordered Segments Matter

text A: "I will inspect the directory"
  -> tool_call: Glob
  -> tool_result: empty
  -> text B: "The directory is empty"

Stored as four ordered records, not one merged AI message.

If text A and text B are merged first, no timestamp sort can put the tool back between them. The engine and runner must preserve the boundary before Node assigns durable positions.

Workspace Transport Choices

SurfaceTransport ModelProtection
Files and Git diffSocket request plus bounded snapshotRoom access, path validation, size and truncation limits
PTY terminalLong-lived session plus byte deltasSession ownership, bounded output tail, controlled input
Dev previewServer-managed preview sessionTarget validation, lifecycle control, viewport limits
Media assetsControlled HTTP asset routeRoom authorization, MIME and size checks

Sandbox and Daemon Recovery

  • One daemon is reused per sandbox, but turns are serialized so two agents do not edit the same workspace concurrently.
  • Daemon startup is locked by sandbox and removes stale runner or Codex child processes before replacement.
  • An active turn extends E2B TTL; completion returns the sandbox to the shorter idle policy.
  • A missing turn_released event cannot block forever because the wait has a timeout and recovery path.
  • On process shutdown, Node reclaims daemons it owns. Durable room, turn, and sandbox identity remain in storage.
  • Artifact upgrades use export, create, import, Git initialization, and an atomic sandbox-ID swap.

Frontend State Machines and Performance

  • The media viewer locks a touch gesture into one mode after crossing a threshold; it does not change interpretation halfway through.
  • Double-tap detection uses both time and distance, while swipe dismissal accepts either sufficient distance or velocity.
  • Gesture handlers write pending values. One animation-frame callback batches track, stage, and image transforms.
  • The sticker picker renders the current page and nearby pages instead of all 2,275 stickers.
  • Optimistic messages move through pending, confirmed, or failed states and reconcile by stable IDs.
  • Older-page insertion preserves scroll position using the change in scroll height.

Persistence and Consistency

  • CompositeRoomStore sends durable operations to Redis or PostgreSQL and realtime presence operations to Redis.
  • The same store contract tests run against both durable implementations to catch behavioral differences.
  • Redis Lua scripts make multi-step membership, version, and delete operations atomic.
  • Room settings use a monotonic version and full-object replacement so stale clients do not merge incompatible fields.
  • PostgreSQL is the source of truth for message pages; Redis cache entries are invalidated only after successful writes.

Technical Challenge Follow-Up Bank

Practice answering the question first, then give the reason and tradeoff. These are intentionally more detailed than the quick cards.

Challenge 1 follow-up: Why not run the agent directly inside the Fly.io app?

The application server holds trusted state and credentials. Agent commands may install packages, consume CPU, open ports, or execute untrusted repository code. Running both in one process would increase the blast radius of a bad command.

E2B provides an isolated execution boundary. Node still owns identity, permissions, durable turns, billing, and lifecycle, so the sandbox cannot grant itself access to another room.

Challenge 2 follow-up: Why not sort tool events by timestamp in the browser?

The old runner could merge text before and after a tool into one assistant message. The browser could move that whole message, but it could not insert the tool into the lost internal boundary. Also, timestamps can tie or arrive late.

The fix belongs where execution order is first known: the engine emits ordered boundaries, the runner forwards them, and Node persists a monotonic position.

Challenge 3 follow-up: Why not place the complete room transcript in every prompt?

It wastes context tokens, repeats irrelevant content, and may expose internal message fields. Instead, the agent requests bounded history or search results only when needed through the RoomTalk CLI.

The runner keeps the real upstream endpoint and token behind a per-turn local broker. The server rechecks room, client, turn, and mode, so possession of a CLI command is not enough to read arbitrary data.

Challenge 3 follow-up: How is Plan mode prevented from modifying the workspace?

Plan shell uses a read-only mount namespace and blocks direct network access while allowing only the turn's local broker socket. This is stronger than maintaining a list of supposedly safe shell commands, because Python, redirects, and command composition could bypass an incomplete allowlist.

Challenge 4 follow-up: Why do files, PTY, and previews use different protocols?

A file read is a bounded request with one response. A PTY is interactive and produces an ongoing byte stream. A preview has a target, viewport, lifecycle, and potentially screenshots or recordings. Forcing them into one transport model would either lose capabilities or create an overly complex protocol.

Challenge 5 follow-up: Why is the daemon registry kept in Node memory?

The registry contains live stdin and stdout handles owned by the current Node process. Those operating-system handles cannot be serialized into PostgreSQL and restored by another process.

Room, turn, and sandbox identity are durable. Live handle ownership is recovered through startup cleanup, health checks, bounded waits, and shutdown reclamation. Reconnect metadata could become durable later if E2B provides stable reconnectable command handles.

Challenge 6 follow-up: Why use a gesture state machine?

The same finger movement can begin as a tap, pan, page swipe, or dismiss gesture. The state machine delays the decision until movement crosses a threshold, chooses a mode from direction and zoom state, and then locks it. Locking avoids a diagonal gesture switching behavior midway.

Pinch zoom also preserves the finger midpoint, and animation work is batched through requestAnimationFrame to avoid repeated layout and paint work.

Challenge 7 follow-up: Why cache by asset ID instead of presigned URL?

A new signed URL may be generated for the same private object, so URL-based keys create duplicate cache entries. The asset ID is stable and represents the logical media object.

In-flight Promise deduplication prevents a thundering herd, and LRU eviction limits browser storage. Object URLs must also be revoked when entries are removed to release memory.

Challenge 8 follow-up: Why is 100vh unreliable on mobile?

Mobile browser chrome and the software keyboard change the visible area without always changing the CSS layout viewport in the same way. Safari and Chrome also behave differently.

RoomTalk reads Visual Viewport height, keeps a stable baseline, publishes CSS variables, and uses composition guards so Enter does not submit while an IME candidate is still being selected.

Challenge 9 follow-up: Why virtual pages instead of rendering every sticker?

Thousands of image elements increase DOM work, memory use, decoding, and network requests. Rendering the current page and nearby pages keeps the active DOM small, while adjacent preloading hides most navigation latency.

Long-press preview uses a timer and suppresses the following click so lifting a finger after preview does not accidentally send the sticker.

Challenge 10 follow-up: How do optimistic messages avoid duplicates?

The client creates a pending message with a stable client-generated identity. The server acknowledgement or broadcast is reconciled with that pending record instead of blindly appended. Failure changes the same item to a retryable state.

Optimistic rendering improves perceived latency, but the server remains authoritative for final content, status, ordering, and permissions.

Challenge 11 follow-up: Why is mobile recovery multi-layered?

A dead mobile connection does not always emit disconnect, and reconnect, visibility, online, and page-restore events vary by browser. RoomTalk checks health when any relevant signal appears, but deduplicates concurrent recovery attempts.

A reconnect indicator is delayed so a fast healthy rejoin does not flash a spinner. Password verification can be reused within the active session.

Challenge 12 follow-up: How could the Redis-to-PostgreSQL migration be rolled back?

The durable store is selected by configuration, and migration does not immediately delete the Redis copy. The migration script is idempotent and supports dry-run, so the team can verify counts and rerun safely.

Realtime presence stays in Redis in either mode. Only the durable implementation changes, which keeps the rollback boundary small.

Challenge 13 follow-up: Why not use CRDTs for room state?

CRDTs are valuable when peers must edit independently and merge offline changes. RoomTalk already has a central authority and most room-setting writes are small, online, server-validated operations.

A monotonic room version, complete-object replacement, and acknowledgement-based read-your-write behavior solve the actual problem with much less complexity.

Challenge 14 follow-up: What belongs inside an AI provider adapter?

The adapter translates model names, request options, provider stream events, error types, token usage, and pricing into RoomTalk's internal contract. Socket handlers should not contain provider-specific branches.

Normalization makes saved AI messages and recovery provider-neutral, while the adapter still preserves provider-specific capabilities when the internal model supports them.

Challenge 15 follow-up: Why use Redis Lua instead of a Node transaction?

Separate Redis commands sent from Node can interleave with commands from another server instance. A Lua script executes the checks and writes atomically inside Redis.

This is useful for room version increments, per-client socket reference counting, and deleting a message while recomputing room activity. The tradeoff is that scripts require dedicated tests and careful maintenance.

Challenge 16 follow-up: Why are shared contract tests necessary?

A TypeScript interface checks method shapes, not behavior. Redis and PostgreSQL may disagree on missing records, ordering, duplicate writes, or pagination boundaries while still compiling successfully.

The same scenario suite runs against both implementations. Database-specific tests are still useful, but contract tests protect the behavior expected by the rest of the application.

Challenge 17 follow-up: Why invalidate instead of updating cached pages in place?

A single message change can affect several cached pages, last-message state, reactions, and ordering. Correctly patching every cached shape is complex and easy to get wrong.

Invalidation is simpler: the next read reloads from PostgreSQL and repopulates the cache. The tradeoff is one slower read after a write, which is acceptable for stronger correctness.

Challenge 18 follow-up: Why estimate tokens instead of using exact tokenization?

Providers and models may tokenize differently, and running every provider tokenizer adds coupling and cost. A conservative estimate is fast and sufficient for selecting history before the request.

Selection first applies semantic truncation for retry or edit, then a message-count limit, then walks backward from newest history under the token budget. Actual provider usage is recorded afterward for billing and reporting.

6. Scalability Discussion

Current Bottlenecks

  • A single small Fly.io VM has limited CPU and memory for Socket.IO connections and AI stream coordination.
  • Large room fan-out creates more socket broadcasts.
  • AI providers and E2B are external dependencies with their own rate and concurrency limits.
  • Large message histories require pagination and careful frontend rendering.

Horizontal Scaling Plan

Load balancer
  -> Node instance A
  -> Node instance B
  -> Node instance C

All instances share:
  - Redis adapter for cross-instance Socket.IO events
  - PostgreSQL for durable data
  - Redis for presence, sessions, and cache
  - S3/Tigris for media and artifacts

Sticky sessions: A client should return to the same Socket.IO instance during a connection, especially when long-polling fallback is enabled. The Redis adapter distributes room events across instances but does not replace connection affinity.

What Changes at 100,000 Connections?

  • Add more Node instances and measure memory per connection.
  • Shard or partition very large rooms and reduce unnecessary broadcasts.
  • Use backpressure and queues for expensive AI or background work.
  • Track connection count, event latency, Redis load, database latency, provider errors, and sandbox startup time.
Good interview answer: Do not claim the current deployment already handles 100,000 users. Explain the current limit, the next bottleneck, and the measurement you would use before redesigning.

7. Security

The main idea is least privilege: every request receives only the power needed for that specific action.

Identity

The socket registers a stable client ID. If the account has credentials, the server also verifies its password token or Google identity.

Room Authorization

The server loads room membership and role, then checks the exact action: join, post, edit, delete, manage members, or use code-agent tools.

Sandbox Isolation

Untrusted commands run in E2B, not in the Fly.io Node process.

Scoped Tokens

Context, model, and publishing credentials are short-lived and cannot be exchanged for each other.

Private Media

Uploads and downloads use short-lived signed URLs after room-access checks.

Secret Handling

Provider keys remain server-side. Connected Codex credentials are encrypted and injected only for the relevant run.

Why a UUID Alone Is Not Authentication

A clientId identifies a browser, but anyone who learns it could try to reuse it. For protected accounts, RoomTalk requires a verified auth token before binding that identity to the socket. Authorization checks still run for every room action.

Safe Room Context

The agent never receives database credentials. It asks for bounded data through the RoomTalk CLI; the server verifies the turn-scoped capability and removes internal fields before responding.

9. Interview Stories

Story 1: Control Plane / Execution Plane

The key architecture decision was to keep the Node server as the control plane and move all code execution into E2B sandboxes. This made the system safer because the production server coordinates work but does not run untrusted code directly.

Story 2: Redis/PostgreSQL Composite Storage

Redis is great for realtime state, but durable chat history and auth data need stronger persistence. The CompositeRoomStore lets the app use PostgreSQL for durable data and Redis for presence and cache while keeping one interface for callers.

Story 3: Realtime Agent Transcript

A coding agent produces text, tool calls, tool results, usage updates, and final answers. RoomTalk maps these structured runner events into room messages so users can follow the agent's work step by step.

Story 4: Mobile Recovery

Mobile browsers frequently pause tabs and disconnect sockets. RoomTalk tracks the desired active room and repairs membership after reconnect, visibility changes, and page restore events.

10. Common Q&A

Why use Socket.IO instead of plain HTTP?

RoomTalk needs live two-way updates for chat messages, AI streaming, terminal output, room presence, and workspace events. Socket.IO gives realtime communication, room broadcasting, acknowledgements, and reconnection behavior.

Why use E2B instead of running commands on the Node server?

Running user or AI-generated code on the production server would be unsafe. E2B gives each room an isolated sandbox where commands, tests, and dev servers can run without risking the main app server.

What is a PTY terminal?

A PTY terminal is an interactive terminal session. In RoomTalk, the terminal UI is in the browser, but the shell runs inside the E2B sandbox. It supports long-running commands, live output, prompts, and Ctrl+C.

What is the reusable JSONL daemon?

It is a long-running sandbox process. The Node server sends it JSONL requests and receives JSONL events. Reusing it avoids restarting the runner for every turn and makes streaming structured events easier.

How does the agent read room history safely?

The agent receives a short-lived turn-scoped token. It can request bounded room context through server APIs, but the server still checks room, client, turn, and mode before returning safe data.

How do you avoid two agents editing the same workspace at once?

The server tracks active turns per room and rejects another running turn. The sandbox daemon also has an active-run lock, so there is a second layer of protection inside the execution plane.

What happens if Redis goes down?

Realtime state such as online presence and socket sessions may be lost, and clients need to reconnect. Durable data such as rooms and messages can remain in PostgreSQL when PostgreSQL is the durable store.

Why publish static artifacts outside the sandbox?

Sandboxes are temporary. They can pause, expire, or be replaced. Static artifacts are stored in object storage and served from stable URLs, so published output survives beyond the sandbox lifetime.

What happens when a sandbox pauses or its artifact version changes?

An idle sandbox can pause with memory preserved and resume later. For a pinned artifact upgrade, RoomTalk exports a bounded workspace archive, imports it into a new sandbox, restores Git state, and then swaps the stored sandbox ID. Published static sites are separate and remain available.

What would you improve next?

A good answer is to add a durable workspace revision layer. Today Git state and archive migration preserve continuity, but a product-level immutable revision history would make rollback and auditing stronger.

Why Socket.IO instead of raw WebSocket?

Socket.IO already provides rooms, acknowledgements, reconnection, fallback transport, and a Redis adapter. Raw WebSocket gives more control but would require RoomTalk to build those features itself.

Why not use Server-Sent Events for AI streaming?

SSE works well for one-way server-to-browser text. RoomTalk also needs browser-to-server terminal input, interrupts, steering, approvals, presence, and room events, so one bidirectional Socket.IO channel is simpler.

Why not use microservices?

The current scale does not justify the deployment and debugging cost. A modular Node monolith keeps development simple. A separate worker becomes useful only when measured AI or background work becomes a bottleneck.

How do you guarantee message ordering?

The server assigns durable positions. Clients display that order instead of trusting device timestamps, which may disagree across phones and laptops.

How do you reduce database deadlock risk?

Most message writes are append-only inserts, which avoid competing updates to the same row. Room-setting changes use small atomic updates with a monotonic version and return the latest row. Transactions stay short, and retry is available for a real serialization or deadlock failure.

How do you handle large message histories?

The store reads pages instead of the whole room. PostgreSQL queries durable history, Redis may cache recent pages, and the client preserves scroll position while older pages are prepended.

What was a difficult production-style bug?

Mobile background recovery could briefly leave two sockets for one browser and double the member count. Presence was corrected by counting unique client IDs and making room rejoin idempotent.

13. Engineering Maturity

Incremental Migration

Redis to PostgreSQL used an interface, a configuration switch, and rollback instead of one irreversible cutover.

Safety Guards

Persistence smoke tests accept only clearly named test or E2E databases, reducing the chance of touching production.

Docs as Code

Configuration and architecture live beside the code and change through review.

CI Guardrails

Deployment checks required configuration before release instead of waiting for a runtime crash.

Idempotent Operations

Retries and migrations are designed so repeating the same operation does not create duplicate data.

Two-Layer Release Contract

The Fly app and pinned E2B artifact deploy separately. Runner changes require a new artifact version, template build, smoke test, and matching production pins.

Honest Boundaries

Git and archive migration preserve workspace continuity, but they are not yet a complete immutable revision and rollback product.

Final Practice Answer

RoomTalk turns a realtime chat room into a collaborative AI coding workspace. The browser shows chat, files, diffs, terminal, previews, and artifacts. The Node control plane manages users, permissions, messages, AI turns, tokens, storage, cost, and sandbox lifecycle. The E2B execution plane runs untrusted code in isolated per-room sandboxes. A reusable JSONL daemon streams structured agent events back to the room, so users can follow AI text, tool calls, command output, and final results in realtime.
