一、30 秒项目介绍1. 30-Second Pitch

Message System 是一个实时 AI 协作平台，核心是共享房间和沙盒化 Code Agent workspace。Message System 在 Fly.io 上作为 control plane，负责身份、权限、持久化 turn、usage/cost、对象存储和 sandbox lifecycle；每个 Code Agent 房间在 E2B 中拥有隔离的 execution plane，通过可复用 JSONL daemon 运行 Coco（自研 CLI coding agent），也支持用户连接自己的 Codex 订阅后通过 Codex CLI 或 app-server 在房间里工作。浏览器里可以直接查看和编辑文件、审阅 Git diff、写行级 review comment、使用 PTY terminal、打开 dev-server preview，并把静态产物发布成不依赖 sandbox 生命周期的稳定 URL。普通聊天仍支持多 Provider AI streaming、私有媒体、Redis/PostgreSQL 组合存储和移动端恢复。
Message System is a realtime AI collaboration platform centered on shared rooms and sandboxed code-agent workspaces. A Fly.io control plane owns identity, permissions, durable turns, usage/cost, object storage, and sandbox lifecycle; each code-agent room gets an isolated E2B execution plane whose reusable JSONL daemon can run Coco, our self-built CLI coding agent, or Codex through a user's connected Codex subscription via CLI/app-server adapters. In the browser, users can edit files, review Git diffs, attach line-level comments, use a PTY terminal, open dev-server previews, and publish static artifacts to stable URLs independent of sandbox lifetime. Regular chat still includes multi-provider AI streaming, private media, Redis/PostgreSQL composite storage, and mobile recovery.

1. CompositeRoomStore — 双存储分层1. CompositeRoomStore — Dual-Store Layering

这是整个后端最核心的抽象。一个 CompositeRoomStore 组合了三个子存储：

This is the most important abstraction in the backend. A single CompositeRoomStore composes three sub-stores:

  • DurableRoomStore：持久数据（房间、消息、成员、媒体资产、认证）。可以是 Redis 或 PostgreSQL，运行时通过环境变量切换。
  • RealtimeRoomStore：始终是 Redis。管理 socket session、在线成员、临时状态。服务重启后自动重建。
  • RoomMessageCacheStore：可选的 Redis TTL 缓存，在 PostgreSQL 模式下减少数据库读压力。
  • DurableRoomStore: Persistent data (rooms, messages, members, media assets, auth). Can be Redis or PostgreSQL, switched at runtime via environment variable.
  • RealtimeRoomStore: Always Redis. Manages socket sessions, online members, ephemeral state. Rebuilt automatically on restart.
  • RoomMessageCacheStore: Optional Redis TTL cache that reduces database read pressure in PostgreSQL mode.
我们把存储拆成"持久"和"实时"两层。持久层可以在 Redis 和 PostgreSQL 之间切换——开发用 Redis 快速迭代，生产切到 PostgreSQL 保证数据安全。实时层始终用 Redis，负责 session、presence 和 pub/sub。中间的短 TTL message cache 让最近历史优先从 Redis 读取，miss 才查询 PostgreSQL；写入时按 room/version 主动失效，不能把 TTL 当一致性机制。
We split storage into durable and realtime layers. The durable layer can switch between Redis and PostgreSQL — Redis for fast iteration in development, PostgreSQL for production durability. Redis always owns sessions, presence, and pub/sub. A short-TTL message cache serves recent history first and queries PostgreSQL on miss; writes invalidate by room/version, so TTL is never treated as the consistency mechanism.

2. Socket.IO 多实例扩展2. Socket.IO Multi-Instance Scaling

Client A ──WSS──▶ Instance 1 ──Redis pub/sub──▶ Instance 2 ──push──▶ Client B

用 @socket.io/redis-adapter，所有 Socket.IO 事件通过 Redis pub/sub 同步到所有实例。客户端不感知自己连的是哪个实例。

Using @socket.io/redis-adapter, all Socket.IO events are synced across instances via Redis pub/sub. Clients are unaware of which instance they're connected to.

追问：sticky session 为什么必须？

如果允许 polling fallback，Socket.IO 握手会先做 HTTP polling 再升级 WebSocket；这些请求落到不同实例会破坏 session，因此负载均衡需要 connection affinity。也可以强制 websocket-only 来减少这个要求，但会放弃 polling fallback。跨实例房间广播仍由 Redis adapter 处理。

Follow-up: Why are sticky sessions required?

With polling fallback enabled, Socket.IO starts with HTTP polling and upgrades to WebSocket. If those requests land on different instances, the session breaks, so the load balancer needs connection affinity. WebSocket-only transport can reduce this requirement at the cost of losing fallback. Redis adapter still handles cross-instance room broadcasts.

3. AI 流式响应3. AI Streaming Response

  • 消息创建时 status: 'streaming'，完成后改为 'complete'，出错改为 'error'。

  • The server uses the OpenAI SDK's streaming API, reads chunks incrementally, and pushes them to the client via Socket.IO in real time.
  • Messages are created with status: 'streaming', changed to 'complete' on success, or 'error' on failure.
  • On server restart, aiStreamRecovery scans all messages with status = 'streaming' and marks them as 'error' to prevent zombie streams.

追问：为什么不用 SSE？

已经有 Socket.IO 长连接了，再开 SSE 是多余的连接。Socket.IO 事件模型可以区分不同房间、不同用户的 AI 流，比 SSE 灵活。而且 AI 流需要和消息历史、状态同步走同一个通道，不适合单独开一个 HTTP 流。

Follow-up: Why not use SSE?

We already have a Socket.IO persistent connection — opening another SSE connection would be redundant. Socket.IO's event model can distinguish AI streams by room and user, which is more flexible than SSE. Plus, AI streams need to share the same channel as message history and state updates — a separate HTTP stream doesn't fit.

4. 媒体上传流程 — Presigned URL4. Media Upload Flow — Presigned URL

Client Server S3 │ │ │ │── request upload URL ──▶│ │ │ │── generate presigned ──▶│ │◀── presigned PUT URL ──│ │ │ │ │ │──── PUT file directly ─────────────────────────▶ │ │ │ │ │── confirm upload ──────▶│ │ │ │── create MediaAsset ──▶ │ │◀── message with asset ─│ │
文件不经过我们的服务器。客户端先向服务器请求一个临时上传 URL（presigned URL，有效期 15 分钟），然后直接上传到 S3。上传完成后通知服务器，服务器创建 metadata 记录。下载也是类似的，服务器生成临时读取 URL 返回给客户端。这样服务器不处理大文件流量，带宽和 CPU 都省了。
Files never pass through our server. The client requests a temporary upload URL (presigned URL, valid for 15 minutes) from the server, then uploads directly to S3. After upload, it notifies the server, which creates a metadata record. Downloads work similarly — the server generates a temporary read URL. This way the server handles zero file traffic, saving both bandwidth and CPU.

追问：presigned URL 安全吗？

URL 本身包含签名，只对特定 bucket/key 有效，有过期时间。泄露了也只能上传到这一个位置，不能读其他文件。而且我们的 bucket 设置了 block all public access，没有 presigned URL 什么都看不到。

Follow-up: Are presigned URLs secure?

The URL itself contains a cryptographic signature, valid only for a specific bucket/key and with an expiration time. Even if leaked, it can only upload to that one location — it can't read other files. Our bucket has "block all public access" enabled, so nothing is visible without a presigned URL.

6. 权限、Context 与 Scoped Token6. Permissions, Context, and Scoped Tokens

  • 四种模式：Plan、Edit、Approve for me、Full access。Plan 的 Shell 由 OS mount namespace 强制只读，不依赖字符串命令白名单。
  • Model gateway：每个 turn 签发绑定 room/client/turn/provider/model/budget 的短期 token；provider key 不进入浏览器，也不作为 prompt 内容。
  • Room context broker：不把整个房间历史塞进 system prompt；Agent 通过 Unix socket 上的 message-system room history/search/delta/message 按需读取有界 projection。
  • Static publish：可写模式拿到独立 publish token；文件从 E2B 通过 presigned PUT 直传 Tigris，Fly 只验证 metadata 并原子切换 manifest。
  • Four modes: Plan, Edit, Approve for me, and Full access. Plan shell writes are blocked by the OS mount namespace, not a string command allowlist.
  • Model gateway: each turn receives a short-lived token bound to room/client/turn/provider/model/budget; provider keys never enter the browser or prompt.
  • Room-context broker: full room history is not injected into every system prompt. The agent reads bounded projections on demand through message-system room history/search/delta/message over a Unix socket.
  • Static publishing: writable modes receive a separate publish token. E2B uploads bytes directly to Tigris through presigned PUTs; Fly verifies metadata and atomically swaps the manifest.

追问：为什么不用一个万能 token？

因为 context read、model spend 和 public publish 的风险完全不同。拆成独立、短期、绑定 turn 的 capability 后，泄露时 blast radius 有边界，服务端也能分别校验 mode、room ownership、provider/model 和预算。

Follow-up: Why not issue one universal token?

Context reads, model spending, and public publishing have different risk profiles. Separate short-lived, turn-bound capabilities bound the blast radius and let the server independently enforce mode, room ownership, provider/model selection, and budget.

Code Agent 篇Code Agent

难点 1：把聊天房间升级为共享执行环境Challenge 1: Turning a Chat Room into a Shared Execution Environment

问题：Agent 要读写文件、安装依赖、运行测试和启动服务，但这些命令不能进入 Fly 应用进程；同时房间成员、消息、权限、计费和恢复又必须由 Message System 统一管理。

Problem: The agent must edit files, install dependencies, run tests, and start services, but none of that can execute in the Fly application process. Room membership, messages, permissions, billing, and recovery still need one Message System source of truth.

方案：拆成 control plane 与 execution plane。每个房间绑定一个 E2B sandbox；Message System 的 CodeAgentSessionService 解析本轮 mode/backend、创建 durable turn、签发 scoped capability，再把版本化 JSONL request 发给沙盒 daemon。daemon 可以路由到自研 Coco CLI agent，也可以使用当前成员通过 device authorization 连接的 Codex 订阅启动 Codex CLI 或 app-server。

Solution: Split control and execution planes. Each room binds to one E2B sandbox. CodeAgentSessionService resolves mode/backend, creates a durable turn, issues scoped capabilities, and sends a versioned JSONL request to the sandbox daemon. The daemon can dispatch to the self-built Coco CLI agent or launch Codex CLI/app-server with the current member's subscription connected through device authorization.

Browser ──authenticated socket──▶ Message System control plane │ │ room + turn + scoped env ▼ E2B JSONL daemon ├─ Coco: self-built CLI agent └─ Codex: user's subscription via CLI/app-server │ ▼ /workspace + Git + PTY
最重要的不是“用了 E2B”，而是把 ownership 划清楚：Message System 永远不执行不可信命令，E2B 永远不决定房间权限，Agent 永远不拿数据库或 provider secret。
The important part is not merely “using E2B.” It is the ownership boundary: Message System never executes untrusted commands, E2B never decides room permissions, and the agent never receives database or provider secrets.

难点 2：AI 文本与工具事件的真实顺序持久化Challenge 2: Persisting the True Order of AI Text and Tool Events

问题：早期 runner 实时发送 text delta，却在整轮结束后从 transcript replay 工具事件。结果数据库先保存整段 AI 文本，再保存 tool call/result；刷新后“当前目录为空”会出现在真正执行 Glob 之前。

Problem: The early runner streamed text deltas but replayed tool events only after the entire run. The database therefore stored a full assistant message before tool call/result records, so after refresh “the directory is empty” appeared before the Glob command that established it.

方案：Engine 在工具执行前后发 ordered tool_call/tool_result callback；runner 原样转成 JSONL；Node 收到工具边界就结束当前 AI segment，工具之后的文本创建新 segment。数据库 position 成为唯一顺序事实，前端只负责展示。

Solution: The engine emits ordered tool_call/tool_result callbacks around execution. The runner forwards them as JSONL. Node closes the current assistant segment at a tool boundary and starts a new segment for later text. Database position is the ordering authority; the client only renders it.

text(A) → tool_call(Glob) → tool_result(empty) → text(B) → final │ │ │ │ ▼ ▼ ▼ ▼ AI segment A message N+1 message N+2 AI segment B

追问：为什么不能前端按 timestamp 重排？

因为文本 chunk 的工具边界已经丢失，前端无法知道整段 AI 文本中哪句话属于哪个 tool result 之后。修复必须发生在最先知道真实执行顺序的 Engine/runner 层，并由服务端 position 持久化。

Follow-up: Why not sort by timestamp in the client?

Because the tool boundary inside the accumulated text has already been lost. The client cannot know which sentence belongs after which tool result. The fix must start where execution order is known — engine/runner — and be persisted by server positions.

难点 3：共享房间上下文与最小权限同时成立Challenge 3: Shared Room Context with Least Privilege

问题：Agent 需要知道前几轮讨论和其他成员的消息，但把完整 transcript 与长期 token 注入 prompt 会增加 context 成本、泄露内部字段，也让 Plan 模式变相获得网络能力。

Problem: The agent needs prior discussion and other members' messages, but injecting a full transcript and long-lived token into every prompt increases context cost, leaks internal fields, and can accidentally grant network capability in Plan mode.

方案：每轮创建本地 Unix socket broker。Agent 只看到 message-system CLI，可以按需 history/delta/search/message；真实 upstream URL/token 由 runner 保管。服务端再次校验 room/client/turn，并只投影安全字段。Model 与 publish 另用独立 token。

Solution: Start a local Unix-socket broker per turn. The agent only sees the message-system CLI and fetches history/delta/search/message on demand; the runner keeps the upstream URL/token. The server revalidates room/client/turn and projects safe fields only. Model and publish capabilities use separate tokens.

message-system room history --limit 20 --json
message-system room search "deployment failure" --json
message-system site list --json

Plan Shell 使用 bubblewrap 只读 mount namespace，并关闭直接 IP 网络，只放行本轮 broker socket。这样任意 python、重定向或恶意 shell 组合都无法写 workspace，而不是依赖维护不完整的“安全命令列表”。

Plan shell uses a bubblewrap read-only mount namespace, blocks direct IP networking, and permits only the turn broker socket. Arbitrary Python, redirection, or shell composition cannot write the workspace; security does not depend on maintaining an incomplete “safe command” list.

难点 4：让 Workspace 可检查、可审阅，而不绕过 AgentChallenge 4: Making the Workspace Inspectable and Reviewable Without Bypassing the Agent

Files and diffs use request/ack snapshots; PTY and preview automation use sessions plus delta events; large files and media use controlled asset routes. Git baselines and refs make diffs reproducible, while review comments return to the next agent turn as structured context.

难点 5：长任务、Daemon 与 Sandbox 生命周期Challenge 5: Long Tasks, Daemon, and Sandbox Lifecycle

问题：每轮启动 runner 会重复冷启动；后台服务和 terminal 需要跨工具调用存活；E2B idle timeout、Fly 重启和丢失的 turn_released 事件又可能留下旧 daemon 或让房间卡在 running。

Solution:

Solution:

  1. 整体替换而非合并：服务端返回完整的 room 对象，客户端收到后整体替换本地状态，不做字段级 merge。

房间设置是低频操作，last-write-wins 够用。CRDT 适合高频并发编辑（协同文档），引入复杂度不值得。我们的 version 号保证了排序，整体替换保证了最终一致。

Room settings are low-frequency operations — last-write-wins is sufficient. CRDTs are for high-frequency concurrent editing (collaborative documents); the added complexity isn't justified. Our version number guarantees ordering, and full replacement guarantees eventual consistency.

  3. active Code Agent room 需要 distributed lease，避免两个 control-plane 实例同时控制同一个 sandbox
  4. 聊天消息继续 durable outbox/批处理；sandbox queue 与 chat throughput 分开扩展
  1. Load-test the real payload, TLS, and heartbeat profile to establish per-instance connection/CPU/memory limits instead of relying on a fixed rule of thumb
  2. Scale socket instances horizontally; sticky connections + Redis adapter preserve cross-instance room broadcast
  3. Use a distributed lease for active code-agent rooms so two control-plane instances cannot control one sandbox
  4. Keep durable chat outbox/batching separate from sandbox queue throughput so the two workloads scale independently

为什么用 E2B，而不是在 Fly 容器里直接跑 Docker 或 Shell？
Fly 进程拥有数据库、Redis、对象存储和用户 session 等生产权限，不应该同时执行模型生成的不可信命令。E2B 提供独立生命周期、文件系统、进程和网络边界。Message System 只通过窄协议控制 sandbox，即使 Agent 命令失控，也不会直接进入 control plane。
Why E2B instead of running Docker or shell directly in the Fly container?
The Fly process owns production database, Redis, object-storage, and user-session privileges, so it must not also execute model-generated commands. E2B provides an independent filesystem, process, network, and lifecycle boundary. Message System controls it through a narrow protocol, keeping failures out of the control plane.
为什么需要 sandbox daemon，而不是每轮启动一次 runner？
每轮进程适合 MVP，但会重复加载 Python/Agent runtime，也很难拥有 terminal、后台服务和 backend thread 等跨 turn 状态。一个 sandbox 一个 daemon 可以复用执行环境，同时仍由 Message System 保证同房间只跑一个 mutating turn。代价是必须处理 stale process、health check、shutdown ownership 和丢失 release signal。
Why use a sandbox daemon instead of starting one runner per turn?
Per-turn processes are useful for an MVP, but repeatedly load the runtime and cannot naturally own terminals, background services, or backend thread state across turns. One daemon per sandbox reuses the environment while Message System still serializes mutating turns. The tradeoff is explicit stale-process cleanup, health checks, shutdown ownership, and lost-release handling.
多个成员同时给 Agent 发消息，怎么避免同时修改 workspace？
房间有一个 active-turn guard，真正执行的 mutating turn 串行化。后续 prompt 持久化为 queue，可以编辑、取消或提升为 steer。这样聊天仍是多人实时的，但共享文件系统不会被两个 Agent turn 并发修改。每个 turn 还持久化实际 mode/backend，retry 不会回退到默认权限。
How do you prevent multiple room members from modifying the workspace concurrently?
A room-level active-turn guard serializes mutating agent work. Later prompts become durable queue entries that can be edited, canceled, or promoted into steering input. Chat remains realtime and multi-user, while the shared filesystem never has two agent turns writing concurrently. Each turn persists its resolved mode/backend so retry cannot fall back to default permissions.
Agent 如何看到历史，又不会拿到整个数据库或长期 token？
Runner 为本轮创建 Unix socket broker，只把 message-system CLI 暴露给 Agent。CLI 的 history/search/delta/message 请求由 broker 带 scoped token 转发；服务端重新校验 room/client/turn，并只返回安全 projection。Model spend 与 static publish 使用另外两种 token，不能互相替代。
How can the agent read history without receiving database access or a long-lived token?
The runner creates a per-turn Unix-socket broker and exposes only the message-system CLI. The broker forwards history/search/delta/message requests with a scoped token; the server revalidates room/client/turn and returns a safe projection. Model spending and static publishing use separate non-interchangeable tokens.
Sandbox 暂停或升级后，文件和网页会怎样？
默认 idle 五分钟后 E2B pause，并启用 keep-memory + autoResume，所以同一个 sandbox 恢复时文件和进程内存可以继续。固定 artifact 版本变化时，Message System 会 bounded export/import workspace archive 后原子替换 sandbox。公开静态站点单独存入 Tigris manifest/version，不依赖 sandbox 存活。
What happens to files and published pages when a sandbox pauses or upgrades?
The default E2B policy pauses after five idle minutes with keep-memory and autoResume, so the same sandbox can continue. When the pinned artifact changes, Message System performs a bounded workspace archive export/import and atomically swaps sandboxes. Public static sites live in separate Tigris manifests/versions and do not depend on sandbox lifetime.
为什么用 Socket.IO 而不是原生 WebSocket？
Socket.IO 提供了开箱即用的自动重连、房间/命名空间、ack 回调、Redis adapter 多实例广播。原生 WebSocket 这些都要自己实现。对于聊天场景，Socket.IO 省了大量样板代码。
Why Socket.IO instead of raw WebSocket?
Socket.IO provides auto-reconnection, rooms/namespaces, ack callbacks, and Redis adapter for multi-instance broadcast out of the box. With raw WebSocket, you'd implement all of these yourself. For a chat scenario, Socket.IO saves enormous amounts of boilerplate.
为什么不用微服务架构？
当前规模不需要。单体的开发、部署、调试效率远高于微服务。如果将来 AI 处理成为瓶颈，可以把 AI worker 拆出来用队列解耦，但现在拆了是过度设计。
Why not a microservices architecture?
The current scale doesn't justify it. A monolith is far more efficient for development, deployment, and debugging. If AI processing becomes a bottleneck in the future, we can extract an AI worker and decouple with queues — but splitting now would be over-engineering.
Redis 挂了怎么办？
实时状态（在线成员、session）丢失，用户需要重连，服务端重建。持久数据在 PostgreSQL，不受影响。消息缓存丢失只是多查一次 PostgreSQL，有性能影响但不丢数据。
What happens if Redis goes down?
Realtime state (online members, sessions) is lost — users need to reconnect, and the server rebuilds it. Persistent data lives in PostgreSQL and is unaffected. Message cache loss just means an extra PostgreSQL query — performance impact but no data loss.
数据库死锁怎么处理？
当前消息写入是 append-only（INSERT），不存在行锁竞争。房间设置更新用 room_version 行级版本号 + UPDATE ... RETURNING 原子操作，失败重试。PostgreSQL 的 MVCC 天然避免了读写阻塞。
How do you handle database deadlocks?
Message writes are append-only (INSERT), so there's no row-lock contention. Room settings use room_version row-level versioning + UPDATE ... RETURNING atomic operations with retry on failure. PostgreSQL's MVCC naturally avoids read-write blocking.
如何保证消息顺序？
每条消息有 server-side timestamp 和 position（单调递增）。客户端按 position 排序显示。不依赖客户端时间戳，因为多设备间的时钟可能不同步。
How do you guarantee message ordering?
Every message has a server-side timestamp and a monotonically increasing position. The client sorts by position. We don't rely on client timestamps because clocks across devices can be out of sync.
如果让你重新设计，会改什么？
三件事：1) 增加 Message System-owned immutable workspace revision 层，把 rollback 从当前 sandbox Git 状态提升为可持久恢复的 product contract；目前 archive migration 保证连续性，但不是通用 revision history。2) 如果 E2B 提供稳定 command reconnect handle，就把可重连 daemon metadata 持久化，减少 Fly restart 后的 cleanup/restart。3) 继续拆分 MessagePage 与 workspace orchestrator，把跨 panel session state 收敛到更明确的 store，而不是增加更多 props。
If you were to redesign, what would you change?
Three things: 1) Add a Message System-owned immutable workspace revision layer so rollback becomes a durable product contract rather than current-sandbox Git state; archive migration preserves continuity but is not general revision history. 2) Persist reconnectable daemon metadata if E2B exposes stable command handles, reducing cleanup/restart after Fly restarts. 3) Keep decomposing MessagePage and the workspace orchestrator so cross-panel session state converges on explicit stores instead of deeper prop chains.
怎么处理大量消息的性能？
三个层面：1) 存储层：消息分页加载（readMessagePageByRoom，默认 80 条一页），不一次性加载全部历史。2) 渲染层：React 列表虚拟化可以进一步优化（当前未做，是已知改进点）。3) 缓存层：Redis RoomMessageCacheStore 缓存最近消息，cache hit 避免查 PostgreSQL。
How do you handle performance with a large volume of messages?
Three levels: 1) Storage: paginated message loading (readMessagePageByRoom, 80 messages per page by default) — never loads full history at once. 2) Rendering: React list virtualization could be added (not yet implemented — known improvement). 3) Caching: Redis RoomMessageCacheStore caches recent messages, cache hits avoid PostgreSQL queries.
项目里遇到最难的 bug 是什么？
移动端切后台回来后成员数翻倍。原因是 visibility change 触发重连，重连后 re-join 房间，但旧的 socket 还没断开，服务端认为是两个不同的连接。修复方法是在 updateRoomMemberCount 里按 clientId 去重，同一个 clientId 的多个 socket 只算一个成员。
What was the hardest bug you encountered?
Member count doubling after backgrounding on mobile. The cause: visibility change triggered a reconnect, which re-joined the room, but the old socket hadn't disconnected yet — the server counted two separate connections. The fix was deduplicating by clientId in updateRoomMemberCount — multiple sockets from the same clientId count as one member.

7. 明确承认边界

7. Explicit Boundaries

Git plus archive migration preserves live workspace continuity, but it is not yet an immutable revision/rollback product layer. Clearly separating shipped behavior, recovery mechanisms, and next architecture steps is more credible than claiming everything is complete.
